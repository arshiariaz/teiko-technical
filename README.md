# Teiko Bio — Clinical Trial Immune Cell Analysis

End-to-end analysis of how **miraclib** affects immune-cell population frequencies in melanoma clinical trial patients.

Dashboard link: [https://huggingface.co/spaces/arshiariaz/teiko-dashboard](https://huggingface.co/spaces/arshiariaz/teiko-dashboard)

---

## Quick Start

```bash
# 1. Install dependencies
make setup

# 2. Build the database and run all analyses
make pipeline

# 3. Launch the dashboard (http://localhost:8050)
make dashboard
```

All commands require Python 3.10+ on your `PATH`.

## Database Schema

```sql
CREATE TABLE projects (
    project_id TEXT PRIMARY KEY
);

CREATE TABLE subjects (
    subject_id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL REFERENCES projects(project_id),
    condition  TEXT NOT NULL,
    age        INTEGER,
    sex        TEXT,
    treatment  TEXT,
    response   TEXT
);

CREATE TABLE samples (
    sample_id                 TEXT PRIMARY KEY,
    subject_id                TEXT NOT NULL REFERENCES subjects(subject_id),
    sample_type               TEXT NOT NULL,
    time_from_treatment_start INTEGER NOT NULL
);

CREATE TABLE cell_counts (
    sample_id  TEXT PRIMARY KEY REFERENCES samples(sample_id),
    b_cell     INTEGER NOT NULL,
    cd8_t_cell INTEGER NOT NULL,
    cd4_t_cell INTEGER NOT NULL,
    nk_cell    INTEGER NOT NULL,
    monocyte   INTEGER NOT NULL
);
```

### Design Rationale

The schema is normalized to **Third Normal Form (3NF)**. Each fact is stored exactly once, in the table that represents its natural entity.

| Table | What it models | Why separate |
|---|---|---|
| `projects` | A clinical study (prj1, prj2, …) | Natural grouping entity; room to add PI names, start dates, IRB numbers without touching other tables |
| `subjects` | A unique patient | Age, sex, condition, treatment, and response are properties of the patient — not of individual sample draws. Storing them once avoids update anomalies |
| `samples` | A single biological draw | Sample type (PBMC / WB) and time-point belong to the draw, not the patient. One patient may have many draws across time-points |
| `cell_counts` | Raw cell counts per sample | Wide format (one column per population) for the five known populations. Straightforward to extend with `ALTER TABLE ADD COLUMN` for new markers |

Referential integrity is enforced through `FOREIGN KEY` constraints, preventing orphaned records.

### How it Scales

| Growth axis | How the schema handles it |
|---|---|
| Hundreds of projects | New rows in `projects`; no structural change |
| Thousands of subjects per project | `subjects` grows linearly; `project_id` indexed for fast cohort queries |
| Thousands of samples per subject | `samples` grows linearly; `subject_id` FK indexed |
| New cell populations | `ALTER TABLE cell_counts ADD COLUMN new_pop INTEGER` — backward compatible, no data loss |
| Multiple assay types (CyTOF, scRNA-seq) | Add `assay_type TEXT` to `samples`; filter in queries |
| Longitudinal analyses | `time_from_treatment_start` is a first-class column; range queries are efficient with an index |
| Analytics at scale | All filter axes (condition, treatment, time, sample_type, response) sit on narrow, indexed columns — complex GROUP BY and cohort queries stay fast |

## Code Structure and Design Rationale

### `load_data.py`

Single responsibility: schema DDL and one-pass CSV ingestion.

- Drops and recreates all tables on each run (idempotent — safe to re-run)
- Uses `pandas.DataFrame.to_sql` for bulk inserts rather than row-by-row `INSERT`
- De-duplicates projects, subjects, and samples in memory before writing
- Prints row counts after each table as a built-in sanity check
- `NULL` response values are preserved for healthy controls who have no treatment response

### `analysis.py`

Three self-contained functions, one per analysis part, each accepting a database connection and returning a `pandas.DataFrame`. No global state; `main()` wires them together sequentially.

- **Part 2** (`build_frequency_summary`): melts the wide `cell_counts` table to long format, computes per-sample totals and relative frequencies
- **Part 3** (`get_filtered_merged`, `run_statistical_tests`, `plot_boxplots`): filters to the melanoma / PBMC / miraclib cohort, runs Mann-Whitney U (non-parametric, no normality assumption), saves results and a boxplot PNG
- **Part 4** (`baseline_subset_analysis`): answers descriptive questions about the day-0 baseline cohort via SQL GROUP BY queries, including the average B cell count for melanoma male responders

### `app_dash.py`

A Dash / Plotly dashboard with URL routing and four pages.

- **Sidebar navigation** — clean Inter font, red active-state indicator
- **Overview** — headline metrics, condition/treatment distribution charts, mean population frequencies, sample timeline
- **Statistical Analysis** — interactive Plotly boxplots (responders `#E8312A`, non-responders `#ABABAB`), significance annotations, color-highlighted stats table
- **Subset Explorer** — summary cards and bar charts for the baseline cohort, including the average B cell count for melanoma male responders
- **Data Browser** — filterable, paginated full-data table with dropdown filters for condition, treatment, sample type, and project
- All data is loaded once on first access and cached in a module-level dict — no repeated DB queries on re-render

## Statistical Findings (Part 3)

Mann-Whitney U test (two-sided, α = 0.05) comparing relative cell-population frequencies between responders and non-responders in the melanoma / PBMC / miraclib cohort:

- **CD4 T Cell** is the only population reaching significance (p = 0.013), with higher frequencies in responders
- All other populations (B Cell, CD8 T Cell, NK Cell, Monocyte) do not reach significance at α = 0.05
- Results are saved to `outputs/statistical_results.csv` and reproduced by `make pipeline`
- Mann-Whitney U was chosen over a t-test given the absence of a normality assumption and the sample sizes involved
